Inline edit vs drawer Body types are simple enough (no logo, no children, just EN/AR + slug + status) that inline-expand edit is faster than opening a drawer. The SUV row demonstrates the expanded state. "+ Add body type" CTA uses the same inline pattern — a new editable row appears at the top of the table.
Used-by column The "Used by" count is read-only (computed from SELECT COUNT(*) FROM listing WHERE bodyTypeId = …) and is the input to the confirm-modal copy on deactivation. Sedan: 38 listings, SUV: 52, Pickup: 19, etc.
Confirm modal on deactivate Toggling Active → Inactive opens ConfirmModalService destructive variant:
"Deactivating Sedan will remove it from the customer-browse filter and the create-listing dropdown. 38 existing listings will keep their Sedan tag and continue to display correctly. Confirm?"
Soft delete only — no hard delete No Delete button on any row. Inactive body types stay in DB to preserve the bodyType field on existing listings. The Wagon row demonstrates: 2 listings still display with the Wagon tag, but new listings can't select it.
Slug uniqueness — global BodyType slug is globally unique (Prisma @unique) — there's no parent. Auto-generates from EN name on create; override allowed via the inline edit form. Duplicate slug → inline error: "A body type with slug 'suv' already exists."
No logo, no AR-only sort Body types are visual concepts (Sedan, SUV) — no logos required. The customer-facing browse filter renders them with shared icon assets owned by the FE (truck icon for Pickup, etc.), so no iconUrl column is needed in the schema. Sort defaults to alphabetical (Convertible … Wagon).
