Inline trim editing Trims are edited in-place — no separate trim page or modal. Click × on any chip to remove (optimistic — Toast on failure). Click "+ Add trim" to open the inline pill input (see Corolla row, currently in edit state with "Apex" typed). Enter saves, Esc cancels.
Trim uniqueness Each trim is unique per model (Prisma unique(modelId, name)). Typing a duplicate shows an inline error inside the chip input: "LE already exists on this model." Cross-model duplicates are allowed (Camry LE and Corolla LE both exist).
Slug uniqueness — per brand Model slug must be unique within Toyota — unique(brandId, slug). So Toyota.camry and Lexus.camry would both be valid (different brands), but two Toyota.camry rows are not. Auto-generated from EN name on create; override via the edit-model drawer.
Confirm modal on deactivate Toggling a model Active → Inactive opens ConfirmModalService destructive variant:
"Deactivating Camry will remove it from the customer browse and the Toyota dropdown in create-listing. 8 existing listings will continue to display correctly. Confirm?"
Soft delete only No delete button on rows. Inactive models stay in DB to preserve referential integrity with existing listings (Hilux row demonstrates: 3 trims grayed, 4 listings still display). To reactivate: toggle back on — no confirm needed.
"Edit model" drawer (not shown) Pencil icon opens a model-edit drawer (same pattern as the brand drawer in 02-brand-edit-drawer.html) with fields: nameEn, nameAr, slug + regenerate, isActive toggle. No logo on models (Prisma schema has no logoUrl). Trims live on the row, not in the drawer.
